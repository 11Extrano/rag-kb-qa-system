## 1. 项目与依赖

- [ ] 1.1 创建项目结构（包/模块划分：文档处理、向量存储、检索匹配、答案生成）
- [ ] 1.2 添加依赖：向量库客户端、嵌入/LLM 调用 SDK，以及配置加载方式
- [ ] 1.3 定义配置项：向量库（LanceDB）路径、嵌入模型/端点、LLM 端点、chunk 是否按标题切分、最大长度、重叠大小

## 2. 文档处理

- [ ] 2.1 实现文档上传接口：接收文件或 URL，持久化原始文档到 MySQL 文档表（doc_id、文件名、上传时间、状态等）
- [ ] 2.2 实现文本清洗：从文档提取文本，规范化空白与编码，产出清洗后全文
- [ ] 2.3 实现文本拆分：第一段可选按标题/结构（Markdown 等）切大块，第二段按段落/句子 + 最大长度 + 重叠切分；片段写入 MySQL 片段表（chunk_id、doc_id、text、可选 metadata）
- [ ] 2.4 串联流水线：上传完成后自动触发清洗与拆分，产出可供向量存储消费的 chunk 列表

## 3. 向量存储

- [ ] 3.1 实现 EmbeddingProvider 抽象及至少一种后端（本地或 OpenAI-compatible API）
- [ ] 3.2 接入 LanceDB（vectordb）：仅写入向量与 chunk_id（不存原文），数据持久化到本地目录
- [ ] 3.3 实现“文本片段 → 向量生成完成”流程：消费文档处理产出的 chunk，调用 EmbeddingProvider 并写入 LanceDB
- [ ] 3.4 实现按 query 向量 top-k 检索接口，返回 [{ chunk_id, score }, ...]

## 4. 检索匹配

- [ ] 4.1 实现检索请求接收：用户问答接口入参校验（必填 question），记录检索请求
- [ ] 4.2 将用户 query 转为 embedding，调用向量存储 top-k 检索，取回 chunk_id 与分数
- [ ] 4.3 根据 chunk_id 从 MySQL 片段表取原文、关联文档表取文件名等，组装为“匹配到的内容”列表（chunk_id、doc_id、text、score、文档名等），按 score 降序供答案生成使用
- [ ] 4.4 无 chunk 时返回空列表并可选提示“知识库为空”

## 5. 答案生成

- [ ] 5.1 实现 LLMProvider 抽象及至少一种后端（本地或远程 LLM API）
- [ ] 5.2 实现提示编排：System 约束“仅根据【参考内容】回答、无法得到答案时说明不编造”；User 为【参考内容】（匹配内容格式化并标注 doc/chunk 来源）+ 用户问题；调用 LLM 生成答案
- [ ] 5.3 实现答案生成完成流程：返回 answer 文本，并附带引用列表（文档名、chunk 等）
- [ ] 5.4 无检索结果时返回友好提示（如知识库为空），不调用 LLM 编造内容

## 6. API 与集成

- [ ] 6.1 管理员 REST API：文档上传、文档列表、文档删除（可选）
- [ ] 6.2 用户 REST API：问答接口（请求 question，返回 answer + 引用片段）
- [ ] 6.3 端到端联调：上传文档 → 自动清洗/拆分/写入 MySQL 与 LanceDB → 用户提问 → 检索 → 取原文 → 生成答案并带引用
- [ ] 6.4 为上传与嵌入/LLM 调用增加超时与简单重试；答案生成失败时兜底（如仅返回检索到的片段）
